## Task 1：GitHub repo 與 PR 工作流

**目標**：公開 repo 上線；完整走過一次「branch → PR → merge」流程。

**為什麼**：計畫書 §7 要求小步提交。單人專案也走 PR 流程，是因為
(1) 公開的 PR 紀錄本身是面試素材；(2) CI 檢查掛在 PR 上才有攔截效果；
(3) 這正是目標公司的日常工作形態。

- [ ] **1.1 登入並建 repo**：`gh auth login`（選 GitHub.com、HTTPS、瀏覽器登入）。
  然後在專案目錄執行 `gh repo create e-wallet --public --source=. --push`。
  意思是：用目前目錄當來源，建立公開 repo 並把 `main` 推上去。
  **驗證**：`gh repo view --web` 開啟網頁，能看到 CLAUDE.md 和 docs/。
- [ ] **1.2 保護 main branch**：GitHub 網頁 → Settings → Branches →
  Add branch ruleset：對 `main` 勾選「Require a pull request before merging」。
  （「Require status checks」等 Task 8 有 CI 之後回來加。）
  **驗證**：直接在本地 `git push` 一個對 main 的 commit，應被拒絕。
- [ ] **1.3 冒煙測試 PR 流程**：
  1. `git switch -c chore/pr-smoke`（建立並切到新 branch）
  2. 隨便修一處文件（例如 README 加一行），`git add` + `git commit`
     （英文祈使句，如 `docs: test PR workflow`）
  3. `git push -u origin chore/pr-smoke`，然後 `gh pr create --fill`
  4. 網頁上按 merge，本地 `git switch main && git pull`
  **驗證**：`git log --oneline` 看得到剛才的 merge。

## Task 2：Go module 骨架

**目標**：`go.mod`、兩個進入點、第一個測試、Makefile、linter 設定。

**檔案**：`go.mod`、`cmd/wallet/main.go`、`cmd/reconciler/main.go`、
`internal/platform/version/version.go`（+ `version_test.go`）、`Makefile`、`.golangci.yml`

**背景概念**：
- **Module** 是 Go 的依賴管理單位，`go.mod` 記錄 module path 與依賴。
  module path 習慣用 repo 的 URL（如 `github.com/<你>/e-wallet`），
  因為 Go 的 import 路徑就是用它當前綴。
- **`cmd/<名字>/main.go`** 是慣例：每個可執行檔一個目錄，裡面只做「組裝」
  （讀設定、把元件接起來、啟動），不放業務邏輯。
- **`internal/`** 是 Go 語言級的存取控制：`internal/` 下的 package
  只有本 module 能 import，外部專案不能。應用程式的代碼幾乎都該放這裡。

- [ ] **2.1 初始化 module**：`git switch -c feat/go-bootstrap`，然後
  `go mod init github.com/<你的帳號>/e-wallet`。
  **驗證**：`cat go.mod` 顯示 module path 和 go 版本。
- [ ] **2.2 寫 `cmd/wallet/main.go`**：
  - 概念：`slog` 是標準庫的結構化日誌（Go 1.21+）。「結構化」= 每行 log 是
    JSON，欄位可被機器解析——生產系統的 log 都長這樣，Phase 3 的可觀測性
    也建立在此之上。
  - 要求：建立 JSON handler 的 logger（查 `slog.NewJSONHandler`，輸出到
    stderr）、用 `slog.SetDefault` 設為預設，然後印一行啟動 log，
    至少含 `service`、`version` 兩個欄位（version 先寫死 `"dev"`，2.3 會抽出去）。
  - 文件：https://pkg.go.dev/log/slog （看 package 開頭的範例即可）
  - **驗證**：`go run ./cmd/wallet` 印出一行 JSON，含你設的欄位。
- [ ] **2.3 同樣寫 `cmd/reconciler/main.go`**（service 欄位改名）。
  **驗證**：`go run ./cmd/reconciler` 同上。
- [ ] **2.4 第一個 TDD 循環——`internal/platform/version`**：
  - 概念：這個 package 之後會在 build 時注入 git SHA，現在先只提供預設值
    `"dev"`。重點不是功能，是讓你完整走一次 TDD：**先寫測試（紅）→
    實作（綠）**，之後所有資金路徑代碼都用這個節奏。
  - 順序：
    1. 先建 `version_test.go`：package 宣告、`import "testing"`、
       一個 `TestVersionDefault` 函式，斷言 `version.Version()` 回傳 `"dev"`。
       測試函式的規則：檔名 `_test.go` 結尾、函式名 `Test` 開頭、參數 `*testing.T`。
    2. 跑 `go test ./internal/...`——**預期失敗**（`version.go` 還不存在，編譯錯誤）。
       看清楚錯誤訊息長什麼樣。
    3. 建 `version.go`，寫最小實作讓測試通過。
    4. 再跑一次——**預期 PASS**。
  - 文件：https://go.dev/doc/tutorial/add-a-test
- [ ] **2.5 回頭讓 `cmd/*` 使用 version package**（import 自己 module 的路徑
  是 `github.com/<你>/e-wallet/internal/platform/version`）。
  **驗證**：`go run ./cmd/wallet` 的 version 欄位仍是 `dev`，但值來自 package。
- [ ] **2.6 `.golangci.yml`**：
  - 概念：linter 是靜態檢查器，抓「編譯器不擋但有問題」的寫法
    （沒檢查的 error、無效代碼、可疑轉型…）。golangci-lint 是聚合器，
    一次跑幾十個 linter，各大 Go 團隊標配。
  - 要求：設定檔啟用 `errcheck`、`govet`、`staticcheck`、`revive`；
    其餘先用預設。文件：https://golangci-lint.run/usage/configuration/
  - **驗證**：`golangci-lint run ./...` 通過（或修到通過——它報什麼就是
    你的第一批 Go 慣用法教材，報錯訊息看不懂就拿來問）。
- [ ] **2.7 `Makefile`**：
  - 概念：Makefile 在這裡當「指令目錄」用——把常用操作固定成短命令，
    本地和 CI 跑**完全相同**的 target，這是可重現性的基礎（spec §1）。
    語法注意：縮排必須是 tab 不是空格；每個 target 前加 `.PHONY:`
    宣告（因為它們不產出同名檔案）。
  - 要求三個 target：`lint`（golangci-lint run ./...）、
    `test`（go test -race ./...，`-race` 開資料競爭偵測，資金系統必開）、
    `build`（go build ./...）。
  - **驗證**：`make lint && make test && make build` 全綠。
- [ ] **2.8 開 PR**：commit message 建議
  `feat: bootstrap Go module with entrypoints and lint/test targets`。
  開完在對話裡叫 Claude review，通過後 merge。

## Task 3：proto 合約與 buf

**目標**：`WalletService` v1 合約定案，lint 通過，產生碼 commit 進 repo。

**檔案**：`buf.yaml`、`buf.gen.yaml`、`proto/ewallet/wallet/v1/wallet.proto`、`gen/**`

**背景概念**：
- **Protobuf** 是跨語言的介面定義語言（IDL）：你在 `.proto` 檔定義訊息和
  服務，工具生成 Go 代碼。計畫書 §3 說「proto 是模組介面的 source of truth」，
  意思是：介面長什麼樣，以 proto 檔為準，Go 代碼是從它生出來的。
- **buf** 是 proto 的工具鏈（Mercari 全線採用）：lint 檢查風格、
  breaking 檢查你有沒有改壞已發布的介面（例如改欄位編號——protobuf 靠
  編號序列化，改了就跟舊 client 不相容）、generate 呼叫 codegen 插件。
- 目錄即 package：`proto/ewallet/wallet/v1/` 對應 package `ewallet.wallet.v1`。
  `v1` 進 package 名，將來有不相容改動就開 `v2`，新舊並存。

- [ ] **3.1 `buf.yaml`**：在 repo 根執行 `buf config init` 生出骨架，然後：
  module 路徑指向 `proto/`、lint 用 `DEFAULT` 規則集、breaking 設定
  `against` 對 git main branch 比。文件：https://buf.build/docs/configuration/v2/buf-yaml
- [ ] **3.2 設計 `wallet.proto`**：**先在紙上/對話裡想過再寫**。
  合約要求（spec §2 定案）：
  - `Money` message：`currency`（string，ISO 4217 如 "JPY"）+
    `minor_units`（int64，最小貨幣單位——不變量：禁止 float）。
  - 五個 RPC：`CreateAccount`、`Deposit`、`Withdraw`、`Transfer`、`GetBalance`。
  - 每個會動錢的 request（Deposit/Withdraw/Transfer）必帶 `idempotency_key`（string）。
  - 你自己決定：每個 RPC 的 request/response 各需要哪些欄位？
    （提示：想「呼叫方拿到 response 後需要知道什麼」——例如轉帳後
    至少要拿到 transaction id 和狀態。）欄位編號從 1 開始連續編。
    每個 message 和 RPC 上方寫英文註解說明用途。
  - 寫完先貼給 Claude review 訊息設計（命名、編號、註解），再往下走。
- [ ] **3.3 lint**：`buf lint`。**預期**：零 issue。它常見的抱怨
  （request/response 命名格式、缺註解）都是 Google API 設計規範，
  報錯訊息本身就是教材。
- [ ] **3.4 `buf.gen.yaml` + 產碼**：
  - 設定兩個插件 `protoc-gen-go`（訊息）與 `protoc-gen-go-grpc`（服務 stub），
    輸出到 `gen/`，開 `managed` 模式讓 go_package 自動對應你的 module path。
    文件：https://buf.build/docs/configuration/v2/buf-gen-yaml
  - Makefile 加 `generate` target（buf generate）。
  - 跑 `make generate` 後打開 `gen/` 下的檔案**讀一遍**：找到你定義的
    message 變成了什麼 struct、service 變成了哪些 interface——
    特別找到 `UnimplementedWalletServiceServer`，Task 4 會用它。
  - **驗證**：`go build ./...` 通過；`git status` 顯示 `gen/` 有新檔案，
    一併 commit（spec §1：產生碼進 repo）。
- [ ] **3.5 PR**：`feat: define wallet v1 proto contract`。

## Task 4：gRPC server 骨架

**目標**：`cmd/wallet` 真的聽 port、掛上（未實作的）service，可被 grpcurl 探索。

**檔案**：新增 `internal/wallet/server/server.go`，修改 `cmd/wallet/main.go`

**背景概念**：
- 生成碼裡的 `UnimplementedWalletServiceServer` 是一個「全部方法都回
  `codes.Unimplemented` 錯誤」的基底。自己的 server type **嵌入**（embed）它，
  之後 Phase 1 一個一個方法覆寫。嵌入是 Go 的組合機制：外層 type 自動
  獲得內層的所有方法。
- **gRPC reflection**：讓 grpcurl 這類工具不用拿 proto 檔就能問 server
  「你有哪些服務」。生產環境內部服務通常開著，方便除錯。
- **graceful shutdown**：收到 SIGTERM（K8s 關 pod 時發的訊號）先停止收新
  請求、等進行中的做完再退出。金流服務被硬殺可能留下半途交易，
  這習慣從第一天養成。

- [ ] **4.1 `internal/wallet/server`**：定義 struct（如 `Server`）嵌入
  `UnimplementedWalletServiceServer`，加一個建構函式 `New()`。先不覆寫任何方法。
- [ ] **4.2 接線 `cmd/wallet/main.go`**：
  - 從環境變數讀 port（`WALLET_GRPC_PORT`，沒設就用 `9090`——查 `os.LookupEnv`）。
  - `net.Listen("tcp", ...)` → `grpc.NewServer()` → 把你的 Server 註冊上去
    （生成碼裡有 `RegisterWalletServiceServer`）→ 註冊 reflection
    （package `google.golang.org/grpc/reflection`）→ log「listening」→ `Serve`。
  - graceful shutdown：用 `signal.NotifyContext`（包 SIGINT、SIGTERM）拿一個
    context，另開 goroutine 等 `ctx.Done()` 後呼叫 `GracefulStop()`。
    這是你第一次寫 goroutine——寫完想想：為什麼 Serve 不能和等訊號在
    同一條 goroutine？想不通就問。
  - 文件：https://grpc.io/docs/languages/go/basics/ 的 server 段落。
- [ ] **4.3 驗證**：
  1. `go run ./cmd/wallet`，另開終端 `grpcurl -plaintext localhost:9090 list`
     → 應列出 `ewallet.wallet.v1.WalletService`。
  2. `grpcurl -plaintext localhost:9090 ewallet.wallet.v1.WalletService/GetBalance`
     → 應回 `code = Unimplemented`（這是**正確**結果——合約在、實作留給 Phase 1）。
  3. Ctrl-C → 程式印關閉 log 後乾淨退出。
  4. `make lint test build` 全綠。
- [ ] **4.4 PR**：`feat: serve wallet gRPC skeleton with reflection`。

MySQL 容器「啟動」到「能接受連線」有幾秒落差，healthcheck 讓依賴它的

- [ ] **5.4 Makefile**：`dev`（`docker compose -f deploy/docker-compose.yml up -d`）

- [ ] **5.5 驗證**：

## Task 6：migration 工具鏈

**目標**：golang-migrate 接通，用 baseline migration 證明 up/down 都能跑。

**檔案**：`db/migrations/000001_baseline.up.sql` / `.down.sql`、Makefile 追加

**背景概念**：schema 變更必須版本化——每個變更一對檔案（`up` 前進 /
`down` 回退），工具在 DB 裡維護 `schema_migrations` 表記錄目前版本。
這讓「任何環境的 DB 長什麼樣」變成可重現的事實，CI 和生產都跑同一組檔案。
真正的表（accounts、ledger_entries…）屬於 Phase 1 的設計產出，
現在只放一個無害的 baseline 把管線打通。

- [ ] **6.1 baseline migration**：`up` 建一個標記表
  `schema_baseline (applied_at TIMESTAMP ...)`；`down` DROP 它。
  檔名格式是 `<序號>_<名字>.up.sql`，工具靠序號排序。
- [ ] **6.2 Makefile**：`migrate-up` / `migrate-down`，呼叫 `migrate` CLI，
  `-path db/migrations`，DSN 形如
  `mysql://root:<密碼>@tcp(127.0.0.1:3306)/wallet`——密碼別寫死在 Makefile，
  想辦法從 `.env` 帶入（提示：Makefile 可以 `include` 或用 shell 讀，
  查到做法後跟 Claude 對一下再定）。
  文件：https://github.com/golang-migrate/migrate/tree/master/database/mysql
- [ ] **6.3 驗證**：`make dev` 起著的狀態下——
  1. `make migrate-up` → DB 裡出現 `schema_migrations`（version=1）和
     `schema_baseline`。
  2. `make migrate-down` → `schema_baseline` 消失。
  3. 再 `make migrate-up` → 回到乾淨的 version=1。
- [ ] **6.4 PR**：`feat: wire golang-migrate with baseline migration`。

## Task 7：Dockerfile

**檔案**：`deploy/Dockerfile`、Makefile 追加

**背景概念**：**multi-stage build**——第一階段用完整的 `golang` image 編譯，

Go 要配合 `CGO_ENABLED=0` 編出純靜態 binary 才能在 distroless/static 上跑。

- [ ] **7.1 寫 Dockerfile**：build stage（`golang:<版本>` →

  想想為什麼順序這樣排？跟 Docker layer cache 有關，答不出來就問）→
  runtime stage（`gcr.io/distroless/static-debian12`，`COPY --from` binary，
  `ENTRYPOINT`）。
- [ ] **7.2 Makefile**：`docker-build`（`docker build -f deploy/Dockerfile -t wallet:local .`）。
- [ ] **7.3 驗證**：
  1. `make docker-build` 成功。
  2. `docker run --rm -p 9090:9090 wallet:local` +
     `grpcurl -plaintext localhost:9090 list` → 列出服務。

## Task 8：CI —— lint / generate-check / test

**目標**：三個 job 掛上 PR，branch protection 開始強制。

**檔案**：`.github/workflows/ci.yml`

**背景概念**：GitHub Actions 的單位是 workflow（YAML）→ job（平行跑）→
step。**service container** 是 job 附掛的容器——test job 掛一個 `mysql:8`，
整合測試就有真 DB 可打（Phase 1 開始吃到）。CI 的每個檢查都呼叫
Makefile target，跟本地一字不差。

- [ ] **8.1 寫 workflow**：`on: [push 到 main, pull_request]`，三個 job：
  - **lint**：checkout → 裝 Go（`actions/setup-go`，開 cache）→
    golangci-lint（官方 action）→ `buf lint` + `buf breaking`
    （`bufbuild/buf-action`，breaking 對 main 比）。
  - **generate-check**：checkout → 裝 Go + buf → `make generate` +
    `go mod tidy` → `git diff --exit-code`（有 diff = 你忘了 commit 產生碼，fail）。
  - **test**：checkout → 裝 Go → services 掛 `mysql:8`（env 給密碼和
    database、開 healthcheck、ports 3306）→ `make test`（DSN 用 env 傳入，
    現在還沒有測試用到 DB，掛好管線就是目的）。
- [ ] **8.2 驗證**：開 PR → Actions 頁三個 job 全綠。故意弄髒一次
  （例如改 proto 不重新 generate 就 push）確認 generate-check 會紅，再修好。
- [ ] **8.3 回補 Task 1.2**：branch protection 把三個 job 設為 required checks。
  **驗證**：PR 頁面顯示「Required」標籤。
- [ ] **8.4 PR**：`ci: add lint, generate-check and test workflows`。

## Task 9：Terraform GCP 基礎（常駐成本 ≈ 0）

**目標**：state bucket、API、Artifact Registry、預算告警、CI 的 keyless 身分。

**檔案**：`deploy/terraform/`（`providers.tf`、`main.tf`、`variables.tf`、
`outputs.tf`、`README.md`，可再拆）

**背景概念**：
- Terraform 宣告「雲上應該存在什麼」，`plan` 算差異、`apply` 執行。
  **state** 記錄「上次 apply 後的實況」，放 GCS bucket（不放本地、不進 git
  ——state 可能含敏感值，且換機器就丟）。
- **Workload Identity Federation（WIF）**：讓 GitHub Actions 用 OIDC token
  換 GCP 短期憑證，**不需要**下載 service account 的 JSON key。
  公開 repo 放 key = 事故；WIF 是 PCA 考點也是 Google 官方推薦做法。
- 雞生蛋問題：state bucket 本身沒辦法由它自己管理，所以用 `gcloud`
  手動建一次，指令記進 README。

- [ ] **9.1 一次性 bootstrap**（指令全部記錄到 `deploy/terraform/README.md`）：
  `gcloud auth login` → 建專案（`gcloud projects create`）→ 綁計費帳戶 →
  `gsutil mb` 建 state bucket + `gsutil versioning set on`。
- [ ] **9.2 Terraform 基礎**：
  - `providers.tf`：google provider + `backend "gcs"` 指向 bucket。
  - 啟用 API（resource `google_project_service`）：`artifactregistry`、
    `container`、`sqladmin`、`pubsub`、`iamcredentials`、`sts`、`cloudbilling`。
  - `google_artifact_registry_repository`：一個 docker 格式的 repo。
  - `google_billing_budget`：金額 300 USD，threshold_rules 0.25 / 0.5 / 0.75。
  - **驗證**：`terraform init` → `terraform apply` → 再跑一次
    `terraform plan` 顯示 `No changes`（宣告收斂了）。
- [ ] **9.3 WIF**：resources `google_iam_workload_identity_pool` +
  `google_iam_workload_identity_pool_provider`（issuer 是 GitHub OIDC，
  attribute condition **限定你的 repo**——這行是安全關鍵，寫完貼給 Claude 看）、
  `google_service_account`（名字如 `ci-push`）只綁
  `roles/artifactregistry.writer`、`google_service_account_iam_member`
  讓 pool 能 impersonate 它。輸出（outputs.tf）provider 全名與 SA email，
  Task 10 要用。文件：https://github.com/google-github-actions/auth#setup
- [ ] **9.4 驗證**：`terraform plan` 乾淨；
  `gcloud artifacts repositories list` 看到 repo；
  Console → Billing → Budgets 看到告警設定。
- [ ] **9.5 PR**：`feat: add GCP baseline Terraform (registry, budget, WIF)`。
  （`.terraform/`、`*.tfstate` 已在 .gitignore，確認沒被 commit。）
